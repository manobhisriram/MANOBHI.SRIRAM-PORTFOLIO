The Problem
  • After company events, image segregation across 1,500+ employees took 13 hours of manual effort per event — a full-time workforce dedicated to sorting, labelling, and distributing photos.
  • No searchable archive of enterprise media existed. Finding a photo of a specific employee from a past event required manual trawl through thousands of files.
  • The process was error-prone, time-consuming, and unscalable as the company grew across 12 countries.
Architecture & Engineering Decisions
  • Multi-model pipeline: DeepFace/FaceNet512 for embedding generation, RetinaFace for high-accuracy face detection, MTCNN as a fallback for low-quality or angled images. Adaptive preprocessing handles group photos, low-light conditions, and varied camera angles.
  • Vector search: Pinecone storing 512-dimensional facial embeddings with cosine similarity. Chose Pinecone over pgvector at this scale for managed indexing, sub-second approximate nearest-neighbour search, and zero operational overhead.
  • Latency optimisation: Concurrent ThreadPoolExecutor batch inference — multiple images processed in parallel, reducing per-event indexing time from hours to minutes. Zero downtime deployment on AWS EC2 with rolling restarts.
  • Privacy-first architecture: Encrypted inference pipelines, zero persistent selfie storage (embeddings only, never raw biometrics), session-scoped tokens, and role-based access controls. Designed to pass enterprise IT security and compliance review from day one.
  • Observability: Full pipeline monitoring via AWS CloudWatch — latency per inference call, error rates, batch throughput, and alert thresholds for any accuracy degradation.
Outcomes
  • Post-event processing reduced from 13 hours to 45 minutes — a 94% reduction in processing time.
  • Eliminated the entire manual image segregation and processing workforce — workforce redirected to higher-value roles.
  • Platform adopted daily by 350+ employees, saving an estimated 200+ person-hours of manual labour per month.
  • 97.3% match accuracy across 50K+ indexed enterprise images in production — maintained consistently across diverse lighting and angle conditions.
  • Cleared full enterprise IT security and compliance review — first AI system at Ascendion to pass biometric data governance requirements.

The Problem
  • Executive mobility across 12 countries involved 13 hours of fragmented manual coordination per trip cycle — coordinating drivers, tracking flight delays, communicating ETAs, and managing last-minute changes across time zones.
  • Multiple disconnected systems (email, phone, WhatsApp, spreadsheets) with no single source of truth for trip state.
  • High error rate and frequent missed pickups due to real-time flight changes not being propagated to ground transport teams in time.
Architecture & Engineering Decisions
  • LangGraph multi-agent system: Two specialised agents — a Flight Intelligence Agent (monitors live telemetry, detects delays, recalculates ETAs) and a Ground Transport Agent (optimises routes, dispatches drivers, manages real-time updates). Shared trip-state memory graph ensures both agents always operate on consistent state.
  • Autonomous lifecycle management: The agent graph owns the complete trip lifecycle from departure to arrival — no human needs to touch the system for a routine trip. Human-in-the-loop only triggered for exceptional events (cancelled flights, driver unavailability).
  • Production RAG pipeline: LangChain + Claude 3.5 Sonnet + Qdrant over 5,000+ vectorised trip records — handles trip context retrieval, preference matching (executive preferences, hotel choices, route history), and conversational Q&A over historical trip data.
  • Real-time integrations: Aviationstack for live flight telemetry (delays, gate changes, cancellations), Google Maps API for route optimisation, Microsoft Teams broadcasting for automatic ETA notifications to executive assistants, Twilio/Firebase for GPS sync with driver mobile app.
  • LLMOps: Full observability via Langfuse — every agent decision, tool call, state transition, and LLM inference logged with latency and token cost. Enables root-cause analysis for any trip anomaly.
Outcomes
  • Replaced a 13-hour manual coordination workflow with zero human intervention — fully autonomous since launch.
  • Adopted as Ascendion's global standard operating platform for executive mobility across all 12 countries.
  • 300+ coordinator hours saved per month across all global offices.
  • Zero missed pickups attributable to flight tracking failures since deployment.
  • Full auditability of every trip decision via Langfuse tracing — satisfying enterprise governance requirements.

The Problem
  • Indian healthcare environments with strict data governance requirements cannot send patient data to cloud LLM APIs — ruling out GPT-4o, Claude, or Gemini for any patient-facing triage.
  • Generic LLMs have poor clinical domain knowledge and no awareness of Indian clinical guidelines (ICMR protocols).
  • Existing triage tools produce black-box outputs — no explainability, no source citations, no safety-critical audit trail for regulatory review.
Architecture & Engineering Decisions
  • LoRA fine-tuning: Fine-tuned Phi-3 Mini 3.8B on 15,000 medical QA examples using LoRA — training loss reduced from 1.92 to 1.21, achieving genuine domain adaptation to clinical triage language. Chose Phi-3 Mini for its balance of capability and local inference performance.
  • Ollama local inference: Zero cloud dependency — entire inference runs on institutional hardware. Patient data never leaves the server.
  • 5-node LangGraph agent with conditional routing: HIGH/EMERGENCY cases trigger parallel OpenFDA drug interaction checks; LOW/MEDIUM cases skip it — reducing average latency by ~30%. Full Pydantic schema validation on every node output; missing severity field triggers automatic corrective retry rather than silently failing.
  • Hybrid retrieval: LanceDB (fully local) with semantic chunking + custom Reciprocal Rank Fusion combining dense and full-text search — context precision improved from 0.65 to 0.85 vs. fixed chunking baseline.
  • Safety-critical CI/CD: DeepEval eval gate in GitHub Actions — any push that degrades faithfulness below 0.60 or causes a severity misclassification is blocked from merging. Treated as a patient safety gate, not a performance metric.
  • Production observability: Langfuse with per-decision audit trail, enabling regulatory review of every triage decision with full reasoning chain.
Evaluation Results (Ragas on curated clinical QA pairs)
  • Faithfulness: 0.92 — answers are grounded in retrieved WHO/ICMR context, not hallucinated.
  • Answer Relevancy: 0.90 — responses directly address the clinical question asked.
  • Context Precision: 0.85 — retrieved chunks are relevant to the query (up from 0.65 with fixed chunking).
  • Context Recall: 0.87 — relevant clinical context is not missed during retrieval.
  • CI/CD gate blocks any regression in faithfulness below 0.60 — severity misclassification treated as a patient safety failure.

System Architecture
  • LangGraph multi-agent with conditional state-based routing: Agent graph dynamically selects notification channels (Slack, Teams, email) based on task priority, employee context, and meeting state. No hardcoded workflow logic — all routing decisions are LLM-driven.
  • LangChain + Groq (LLaMA 3) extraction chains: Structured output parsers extract task owner, deadline, priority, and dependencies — ~90% task extraction accuracy across 500+ tasks. Async chain execution keeps inference latency below 2 seconds.
  • Three RAG use cases on Pinecone: (1) Semantic transcript search — find specific discussions across all past meetings. (2) Conversational Q&A over meeting history — ask questions in natural language. (3) Cross-meeting duplicate task detection — prevents the same action item being assigned twice.
  • LLM evaluation metrics tracked per pipeline run via LangSmith — faithfulness, relevancy, and extraction accuracy monitored continuously, not just at launch.
  • Enterprise-ready integration layer: CRM-ready API design enables enterprise drop-in deployment with zero pipeline restructuring. Deployed on AWS with FastAPI backend and real-time SaaS dashboard.
Engineering Highlights
  • Async chain execution reduces P95 latency to under 2 seconds even on long meeting transcripts.
  • Duplicate task detection across meetings using vector similarity — prevents coordination overhead from repeated action items.
  • Priority-based routing logic: HIGH priority tasks go to both Slack and email; MEDIUM tasks to Slack only; LOW tasks batched into a daily digest.
  • Full activity timeline and overdue tracking in the SaaS dashboard — managers can see task completion rates per meeting without manual tracking.
