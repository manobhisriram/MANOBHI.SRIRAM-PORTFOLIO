The Problem
  • After company events, sorting and distributing 50K+ photos across 1,500+ employees took 13 hours of manual effort per event — a full team dedicated solely to photo management.
  • No searchable archive existed. Finding a photo of a specific employee from a past event required trawling through thousands of unsorted files.
  • The process was error-prone, unscalable, and cost hundreds of person-hours per month across Ascendion's 12-country operation.
What Was Built
  • Multi-model facial recognition pipeline: DeepFace/FaceNet512 for embedding generation, RetinaFace for high-accuracy face detection, MTCNN as fallback for difficult angles and low-light conditions. Adaptive preprocessing handles group photos, varied angles, and poor lighting automatically.
  • Pinecone vector search: 512-dimensional facial embeddings stored with cosine similarity — sub-second approximate nearest-neighbour search at enterprise scale. Chose Pinecone over pgvector for zero operational overhead and managed indexing.
  • Concurrent batch inference: ThreadPoolExecutor parallelises multi-image processing — per-event indexing time dropped from hours to minutes. Zero downtime deployment on AWS EC2 with rolling restarts.
  • Privacy-first biometric architecture: Encrypted inference pipelines, zero persistent selfie storage (embeddings only, never raw biometrics), session-scoped tokens, and RBAC. Designed to pass full enterprise IT security and compliance review from day one.
  • Production observability: AWS CloudWatch monitors latency per inference call, error rates, batch throughput, and fires alerts on any accuracy degradation.
Results
  • Post-event processing reduced from 13 hours to 45 minutes — 94% time reduction.
  • Entire manual image segregation workforce freed for higher-value work.
  • Adopted daily by 1,500+ employees — 200+ person-hours saved per month.
  • 97.3% match accuracy maintained consistently across diverse lighting, angles, and group sizes.
  • First AI system at Ascendion to clear full biometric data governance requirements.

The Problem
  • Contract review is one of the most time-consuming, repetitive tasks in legal work. A junior lawyer spends 2–4 hours per contract at ~₹1,400 per review — and most of that time is pattern recognition that AI can do better.
  • No single LLM has the context window, domain breadth, and specialised capabilities needed for full contract analysis — this demanded a multi-model architecture where each model does what it does best.
  • Legal teams and non-technical users needed a way to query contracts without writing API calls or prompts — which is exactly what MCP servers solve.
What Was Built
  • 4-agent AutoGen pipeline across 5 LLMs: Claude Sonnet 4.5 handles clause extraction (structured output), GPT-4o runs risk classification against a 47-pattern risk library, Gemini 1.5 Pro handles template comparison using its 1M token context window, and Claude Opus 4.5 synthesises the final report. Claude Vision + GPT-4o fallback extracts tables, stamps, handwritten annotations, and redlines from the first 20 pages.
  • Production RAG on Milvus/Zilliz Cloud: 3 collections, OpenAI text-embedding-3-small embeddings, Cohere rerank-v3.0 (top-20 to top-5, +17.7% NDCG@5), Groq Llama3 for fast lookups, 34.2% cache hit rate.
  • Regulatory compliance monitoring: APScheduler with SHA-256 hashing monitors 5 Indian and EU regulatory sources daily. When regulations change, the system automatically maps compliance gaps to active contracts.
  • 4 MCP servers (17 tools): Enables zero-API-knowledge contract querying via Claude Desktop. Non-technical legal staff can ask natural language questions about contracts without writing a single line of code.
  • LLM Judge evaluation: 82.4% overall score on 100 CUAD contracts — correctness 88.7%, calibration 80.5%, completeness 79.3%, actionability 81.2%.
  • Red-teaming: 3 iterations of adversarial testing reduced failure rate from 41% to 9% across 5 attack categories including prompt injection, hallucination fishing, and context manipulation.
  • Full LLMOps: Datadog (ddtrace, LLMSpan APM, 40+ DogStatsD metrics, structlog) + Langfuse prompt replay. FastAPI (15 routes), PostgreSQL/Neon (8 tables, async SQLAlchemy + Alembic), Streamlit frontend, Docker Compose, Render deployment, 46 pytest functions with full LLM API mocking.
Why Multi-Model
  • Claude Sonnet 4.5 excels at structured extraction — it reliably outputs consistent JSON clause data.
  • GPT-4o's function calling is the best in class for complex pattern matching against the 47-risk library.
  • Gemini 1.5 Pro's 1M context window is essential for comparing long contracts against full template libraries without chunking.
  • Claude Opus 4.5 produces the most coherent, well-reasoned synthesis reports for legal professionals.
  • Each model is used only where it has a genuine advantage — cost is kept to $0.0094 per review as a result.

The Problem
  • Enterprises running AI at scale face three compounding problems: runaway costs (routing every query through the most powerful model), compliance risk (PII leaking into LLM prompts), and hallucination (especially on complex, contested queries where a single model picks a side).
  • Existing solutions address one problem at a time. This platform addresses all three in a unified pipeline.
  • At 2,000 queries/day without intelligent routing, an organisation pays ~₹1,40,000/year more than necessary — simply by using Opus when Haiku would suffice.
The 15-Step Pipeline
  • Step 1 — PII Detection: Presidio scans every query for Aadhaar numbers, PAN cards, phone numbers, and email addresses before they touch any LLM API. Violations are redacted and flagged.
  • Steps 2–3 — Query Classification: Groq classifies complexity (simple/medium/complex), risk level, and debate-worthiness — guaranteed JSON output with retry on malformed responses.
  • Step 4 — Semantic Cache: Weaviate semantic cache (0.92 similarity threshold) checks if a near-identical query was answered recently — 67.3% hit rate eliminates most repeated LLM calls entirely.
  • Steps 5–6 — Guardrails & Governance: Pattern-based guardrail matching for policy violations, then human-in-the-loop review for flagged queries.
  • Steps 7–8 — RAG Retrieval: Groq query rewriting for retrieval optimisation, then Weaviate hybrid BM25 + vector search (top-10 candidates).
  • Step 9 — Reranking: Cohere reranking narrows to top-5 most relevant chunks — +29.5% quality improvement over unranked retrieval.
  • Steps 10–11 — Intelligent Routing: LiteLLM routes to Haiku (₹0.08), Sonnet (₹0.52), or Opus (₹2.31) based on complexity classification, with auto-fallback. Long-term user memory is injected for personalised context.
  • Step 12 — 3-Agent Debate (for contested queries): Claude Sonnet (Advocate) argues the strongest case for one position, GPT-4o (Critic) rigorously challenges it, and Gemini 1.5 Pro (Judge, 3 tools) delivers the balanced final verdict. Eliminates single-model bias — 32.9% better decision quality vs single-model baseline.
  • Steps 13–15 — Output, Audit & Observability: Permanent audit trail in PostgreSQL, Langfuse per-stage tracing, and Prometheus metrics for the Grafana dashboard.
Evaluation Results
  • TruLens RAG Triad: Answer relevance 84.7%, context relevance 79.1%, groundedness 82.3%.
  • Decision quality: 32.9% improvement over single-model on complex queries; +71.4% improvement in opposing-perspective consideration.
  • Promptfoo regression testing ensures no pipeline change degrades output quality below baseline.
  • Prometheus + Grafana 8-panel dashboard monitors cost per query, cache hit rate, PII detection rate, model distribution, and latency percentiles in real time.

The Problem
  • Indian healthcare environments with strict data governance cannot send patient data to cloud LLM APIs — ruling out GPT-4o, Claude, or Gemini for any patient-facing triage.
  • Generic LLMs have poor clinical domain knowledge and zero awareness of Indian ICMR protocols.
  • Existing triage tools produce black-box outputs — no explainability, no source citations, no audit trail for regulatory review.
What Was Built
  • LoRA fine-tuning: Phi-3 Mini 3.8B fine-tuned on 15,000 medical QA examples from ChatDoctor-HealthCareMagic-100k using LoRA (r=4, alpha=8). Training loss dropped from 1.921 to 1.219 — genuine clinical domain adaptation. Runs fully on Ollama — patient data never leaves the server.
  • 5-node LangGraph agent pipeline: SymptomParser → SeverityAssessor → ClinicalRetriever → DrugChecker → TriageDecision. Every node has typed Pydantic v2 schema validation — malformed outputs trigger auto-retry (max 3) rather than silently failing.
  • Conditional routing for speed: HIGH/EMERGENCY severity triggers parallel OpenFDA drug lookup. LOW/MEDIUM severity skips it — ~30% latency reduction without sacrificing safety for critical cases.
  • Hybrid RAG on LanceDB (fully local): Semantic chunking with all-MiniLM-L6-v2 (cosine similarity 0.75) + full-text search with custom RRF fusion across 65 WHO/ICMR documents and live OpenFDA API. Context precision improved from 0.65 to 0.85 vs fixed chunking baseline.
  • Safety-critical CI/CD: GitHub Actions eval gate in every PR — any push degrading faithfulness below 0.60 or causing severity misclassification is blocked. Treated as a patient safety gate, not a performance benchmark.
  • Full Langfuse audit trail: Every triage decision logged with full reasoning chain for regulatory review.
Evaluation Results
  • Faithfulness 0.92 — answers are grounded in retrieved WHO/ICMR context, not hallucinated.
  • Answer Relevancy 0.90 — responses directly address the clinical question asked.
  • Context Precision 0.85 — retrieved chunks are relevant (up from 0.65 with fixed chunking).
  • Context Recall 0.87 — relevant clinical context is not missed during retrieval.
  • 5/5 DeepEval assertion tests passed on the full clinical QA evaluation set.
